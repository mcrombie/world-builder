## Phase 1 — Lore integration polish
*Completes the current lore system before building on top of it.*

### 1a. Auto-link suggestions
When a region has no `loreRef` and a lore file is loaded, check whether any lore entry name is a close match to the region name (case-insensitive, partial match). Show a "Suggested: Yunethre (region)" button in the InfoPanel that the user can accept with one click. Reduces the manual linking burden for well-named regions.

Files: `InfoPanel.tsx`

### 1b. Batch-link script
`link_lore.py` — command-line script that reads `saved_maps/azhora.azmap` and `azhora.azlore`, applies the `populate_region_lore.py` name-mapping logic to write `loreRef` fields automatically for all matched regions, then saves. Replaces the hardcoded `LORE_TO_REGIONS` dict with fuzzy matching. A `--dry-run` flag prints what would be linked without writing.

Files: `link_lore.py` (new), replaces part of `populate_region_lore.py`

### 1c. Lore entry cross-reference navigation
In the lore reader, `related:` entries listed in the frontmatter are shown as clickable chips. Clicking a related entry navigates the reader to that entry (even if it belongs to a different region) without leaving the current region's context. A breadcrumb trail keeps track: `Yunethre → Celder → Lizeem River`.

Files: `InfoPanel.tsx` (LoreReader component)

### 1d. Climate paint tool
Add a `climate` tool mode to the Toolbar alongside the terrain paint tool. Lets the user paint climate directly onto hexes with a brush — the same radius/brush system as terrain painting. Useful for manually overriding climate generated by the noise-based system. The climate layer toggle in Layers already renders climate colours; the paint tool makes it an editable layer.

Files: `Toolbar.tsx`, `mapStore.ts` (new `paintClimate` action), `types/map.ts` (add `'climate'` to `Tool`)

## Phase 2 — Azhoran historical simulation
*Grounds the existing simulation engine in Azhoran lore.*

### 2a. Faction scenario file and setup script
`azhoran_scenario.json` defines the historical starting state ~976 years before the War of Seven Sons:

```json
{
  "factions": [
    { "id": "proto_mittoli", "display_name": "Proto-Mittoli", "color": "#4a7c4e",
      "regions": ["Yunethre", "North Celder", "Ovesos", ...] },
    { "id": "elagosi_tribes", "display_name": "Elagosi Tribes", "color": "#6baed6",
      "regions": ["Elagos", "Moros Plain", "Eer"] },
    ...
  ]
}
```

`setup_azhoran_scenario.py` reads the scenario file and writes `faction`, `loreRef`, and `notes` fields into `saved_maps/azhora.azmap`. Extends `populate_region_lore.py`; replaces its hardcoded path.

Files: `azhoran_scenario.json` (new), `setup_azhoran_scenario.py` (new)

### 2b. History recording
New IPC handler `sim:record-history` in `index.ts`. Runs N turns automatically, captures every `SimWorldState` into an array, and writes a `.azhist.json` file:

```json
{ "worldwright_history": true, "scenario": "Age of Origins",
  "turns": [{ "turn": 0, "world_state": {...} }, ...] }
```

UI: a "Record history…" button in `SimulateDialog.tsx` with a turn-count input and a progress indicator.

Files: `map/src/main/index.ts`, `SimulateDialog.tsx`, `map/src/preload/index.ts`, `types/map.ts`

### 2c. History replay viewer
`HistoryPanel.tsx` — loaded when a `.azhist.json` is opened. Replaces the live simulation panel with:
- Turn scrubber (slider 0 → N)
- Play / pause at configurable speed (50–500ms per step)
- Jump-to-turn input
- Geopolitical / Cultural view toggle (regions coloured by owner vs. by dominant tradition)
- Event log for the current turn

The geopolitical view is the existing `HexCanvas` region-colour rendering. The cultural tradition view follows the same pattern as the climate layer: a `CULTURE_COLORS` record keyed by tradition ID renders on hex fills.

Files: `HistoryPanel.tsx` (new), `mapStore.ts` (add `historyFile`, `historyTurn`), `App.tsx`, `types/map.ts`

### 2d. Faction assignment UI
In the InfoPanel region panel, add a "Faction" field that is backed by the scenario's faction list (not a free-text input). A dropdown lists the factions defined in the loaded scenario. The selected faction is written to `region.faction`, which is what `wwmap_to_clashvergence.py` reads.

## Phase 3 — Map editing depth
*Tools that make authoring richer and less tedious.*

### 3a. Road / trade route layer
A new edge type alongside rivers: roads. Roads are drawn hex-edge by hex-edge, same as rivers. A road layer toggle controls visibility. Roads are stored in `MapData.roads: Record<string, 'dirt' | 'paved' | 'imperial'>`. The tactical game design document (`two-scale-game.md`) references road movement bonuses; this layer provides the data.

Files: `types/map.ts`, `mapStore.ts`, `HexCanvas.tsx`, `Toolbar.tsx`

### 3b. Settlement detail
Current settlements are a single name + size. Extend `HexData` with optional `settlementType: 'city' | 'port' | 'fortress' | 'temple' | 'ruin' | 'waystation'`. The InfoPanel shows a type selector when a settlement is named. Each type renders a distinct symbol on the canvas. Ruins in particular appear frequently in lore (Qadimuur, the Ascending Peaks, the canyon petroglyphs) and deserve their own visual marker.

Files: `types/map.ts`, `HexCanvas.tsx`, `InfoPanel.tsx`

### 3c. Multi-region selection and bulk edit
Hold Shift + click regions to select multiple. Selected regions are highlighted together and the InfoPanel shows shared fields (faction, climate, status). Editing any shared field writes it to all selected regions simultaneously. Useful for assigning a faction across a large territory in one action.

Files: `mapStore.ts` (add `selectedRegions: string[]`), `HexCanvas.tsx`, `InfoPanel.tsx`

### 3d. Region search
A keyboard shortcut (e.g. `Ctrl+F`) opens a small search bar that filters regions by name. Selecting a result both selects the region and triggers the existing auto-center pan animation.

Files: `App.tsx` or a new `RegionSearch.tsx` dialog

### 3e. Minimap click-to-navigate
The existing minimap in `RandomMapDialog.tsx` is preview-only. Add a persistent minimap panel (toggleable, rendered small in a corner of HexCanvas or as a layer) that shows the full map at low resolution. Clicking the minimap pans the main view to that location with the same smooth animation used for region selection.

Files: `HexCanvas.tsx` or a new `Minimap.tsx` overlay

## Phase 4 — Export and publication
*Getting content out of the editor.*

### 4a. Map image export
"Export as image…" in the File menu renders the current map view (or the entire map at a chosen resolution) to a PNG. Uses an offscreen canvas at higher resolution than the display canvas. Optionally overlays region labels, rivers, and settlements.

Files: `App.tsx`, new `export.ts` utility, `map/src/main/index.ts` (IPC for file save dialog)

### 4b. Lore document export
"Export lore…" generates a single structured HTML or Markdown document from all regions with linked lore entries: region name → linked entry body, sorted geographically (by map centroid). Useful for printing a world atlas or sharing with collaborators. Uses the same `LoreBody` rendering logic already in the InfoPanel.

Files: new `export_lore.py` or in-app TypeScript export

### 4c. Web viewer (read-only)
A browser build of the app (the `build:browser` script already exists) deployed as a static site. Read-only: no Electron IPC, no file saving, no simulation. The lore file and a chosen `.azmap` are bundled at build time. Visitors can pan/zoom the map, click regions to read linked lore, and switch view modes. Useful for sharing Azhora with readers or collaborators.

Files: `vite.browser.config.ts` (already exists), browser-mode stubs in `fileIO.ts`

## Phase 5 — Simulation depth
*Making the simulation historically meaningful.*

### 5a. Cultural tradition layer
Extend `wwmap_to_clashvergence.py` and the Clashvergence output to include `cultural_seeds` per region. The simulation engine tracks cultural composition alongside territorial ownership. The world-builder renders a "culture view" layer: each hex filled with a colour keyed to the dominant tradition in its region. Toggling between geopolitical and cultural view in the replay viewer shows both histories simultaneously.

### 5b. 976-turn calibration
The Azhoran scenario is designed to naturally produce War-of-Seven-Sons conditions at turn 976. Achieving this requires running many recorded histories, comparing faction standings at turn 976 to the lore (Ambron dominant in northeast, Izol controlling republican city-states, Marosh holding the desert peninsula), and adjusting `azhoran_scenario.json` starting values. A `calibrate_scenario.py` script automates batch runs and produces a report.

Files: `calibrate_scenario.py` (new), `azhoran_scenario.json` (tuning)

### 5c. Narrative event log
The simulation's `recent_events` list currently shows abstract event types (expand, attack, develop). Extend the event schema to include narrative descriptions drawn from lore: when Elagosi Tribes expands into Moros Plain, the event reads "The Elagosi lake-fleet secures the Ros estuary" rather than "Faction2 expands to Moros Plain". Event descriptions are generated by matching faction + terrain + region lore to a template library.

### 5d. War of Seven Sons scripted scenario
A specific scenario file (`war_of_seven_sons.json`) sets the starting state at turn 976 with factions matching the novella: Ambron (imperial, large treasury, controls Elagos and northeast), Izol (republican, controls city-states), Solis (Pyrosi), Marosh (desert kingdom), and the Mittoli heartland fragmented. The scenario includes scripted triggers: at turn 0 (= historical year 976), Maro's flight from Ambron is noted as an event; at turn 5, the Vote of Condemnation fires; at turn 10, armies mobilise. The simulation then proceeds without scripting.

## Phase 6 — Two-scale game (long term)
*The tactical / strategic game outlined in `two-scale-game.md`.*

### 6a. Strategic layer
Army stacks on regions, income from terrain tags, movement orders, diplomatic status between factions. Turn structure: income → orders → resolution → upkeep. Builds on the existing region graph and faction assignment system. New `gameState` object lives alongside `MapData` without embedding game data in the map file.

### 6b. Tactical battles
When two army stacks contest a region, zoom to that region's hex grid for a turn-based battle. Entry hexes derived from the border shared with the attacking region. Units placed on hexes, terrain effects from the existing terrain type system. Victory condition: attacker controls the settlement hex (or centroid) after N rounds.

### 6c. Save/load game state
`.azgame` file format: map path + faction state + army positions + diplomatic graph + full history. Separate from `.azmap` (world data) and `.azhist.json` (simulation replay).

## Maintenance backlog

These are not features but quality-of-life fixes worth doing before they compound:

- **`populate_region_lore.py` path** — still targets the old `map/resources/examples/azhora.wwmap` path; replaced by `setup_azhoran_scenario.py` but the old script should be updated or deleted to avoid confusion.
- **`wwmap_to_clashvergence.py` legacy terrain tags** — still references removed terrain types (`tundra_hills`, `desert`, `mediterranean`, etc.). Update the `TERRAIN_TO_TAGS` dict to the current 13-type terrain system.
- **`azhora_lore_map/azhora.azmap`** — archived copy of the map inside the lore directory. Either delete it or document clearly that it is not the working file (`saved_maps/azhora.azmap` is).
- **Climate layer visibility** — the climate layer checkbox is in Layers but does not yet have a legend. A small colour-key overlay in the corner of HexCanvas would make it useful at a glance.
- **InfoPanel region tile count** — currently iterates all hexes on every render. Cache this in mapStore or compute it lazily.
